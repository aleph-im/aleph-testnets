# Aleph Testnets — Design Spec

## Purpose

A repository that deploys Aleph testnets and runs integration tests to verify that the latest versions of each component (pyaleph CCN, aleph-vm CRNs, scheduler, Aleph Rust CLI) work together. Maintainers open PRs to bump component versions in a manifesto; CI validates the change; merging deploys to a permanent testnet.

## Phasing

| Phase | Components Deployed | Tests | New in Manifesto |
|---|---|---|---|
| **Phase 1 (MVP)** | CCN (pyaleph + deps + Anvil) | File upload/download, posts, aggregates, messages | pyaleph, p2p-service, aleph-cli, infra images |
| **Phase 2** | + Credits API, + Nodestatus, + Scheduler | Funded account scenarios, node registration/staking | aleph-api-credit, aleph-nodestatus, scheduler images |
| **Phase 3** | + 2-3 CRNs (DigitalOcean ephemeral for PRs) | Instance create, SSH, lifecycle, full stack | aleph-vm .deb URL, CRN config |

Only phase 1 is implemented now. The design accommodates all three phases.


## Manifesto Format

```yaml
# manifesto.yml
components:
  pyaleph:
    image: alephim/pyaleph-node
    tag: "0.10.1-rc0"

  p2p-service:
    image: alephim/p2p-service
    tag: "0.1.4"

  aleph-cli:
    version: "0.7.1"
    url: "https://github.com/aleph-im/aleph-rs/releases/download/v0.7.1/aleph-cli-linux-x86_64"

infrastructure:
  postgres:
    image: postgres
    tag: "15.1"
  rabbitmq:
    image: rabbitmq
    tag: "3.11.15-management"
  redis:
    image: redis
    tag: "8.4.0"
  ipfs:
    image: ipfs/kubo
    tag: "v0.37.0"
  anvil:
    image: ghcr.io/foundry-rs/foundry
    tag: "stable"
```

**Components** are the software under test (PRs bump these). **Infrastructure** contains companion services that change infrequently. The deploy script parses the manifesto, generates a `.env` with image tags, and `docker-compose.yml` references them via variable substitution (e.g., `${PYALEPH_IMAGE}:${PYALEPH_TAG}`).

The CLI binary is downloaded separately during test setup based on its URL in the manifesto.

## Deployment Pipeline

### On merge to main (`deploy.yml`)

1. Parse `manifesto.yml`.
2. Generate `.env` from manifesto image tags + GitHub Actions secrets.
3. SCP `.env`, `docker-compose.yml`, and `config.yml` to the testnet server.
4. SSH into the server and run:
   ```bash
   docker compose down -v   # stops containers and removes all project volumes
   rm -rf /var/lib/pyaleph/storage/*
   docker compose up -d
   ```
5. Poll CCN API endpoint `GET /api/v0/version` until it returns HTTP 200 (timeout: 120s, poll every 5s).
6. Report success/failure.

### What persists across deployments

- **P2P identity key** — stored on the host at `/etc/pyaleph/keys/` and bind-mounted into the container. This directory is outside Docker volumes and the pyaleph storage path, so it is never wiped. On first deploy, the key is auto-generated by pyaleph on first start.
- **config.yml** — regenerated on every deploy from the template, SCP'd to the server.

### What gets wiped

`docker compose down -v` removes all named Docker volumes (postgres, redis, rabbitmq, ipfs data). The `rm -rf /var/lib/pyaleph/storage/*` wipes the CCN's file storage (bind-mounted). Together this gives a clean slate while preserving the P2P identity key (which lives at `/etc/pyaleph/keys/`, outside any volume or storage path).

### Secrets (GitHub Actions)

- `TESTNET_SSH_PRIVATE_KEY` — SSH key for the testnet server
- `TESTNET_HOST` — IP/hostname
- `TESTNET_USER` — SSH user

## CCN Stack

The `docker-compose.yml` is based on pyaleph's existing `deployment/samples/docker-compose/docker-compose.yml`, adapted to:

- Use image tags from the `.env` (fed by manifesto).
- Include an Anvil container as the Ethereum endpoint (no real chain sync needed).
- Use `ALEPH-TESTNET` as the queue topic for isolation.
- Disable unnecessary chain connectors (BSC, Solana, etc.).

Services:
- `pyaleph` — core CCN node
- `pyaleph-api` — API server
- `postgres` — message database
- `rabbitmq` — P2P message broker
- `redis` — cache
- `ipfs` — content storage (Kubo)
- `p2p-service` — Rust P2P daemon
- `anvil` — local Ethereum node

## pyaleph Configuration

The `config.yml.tpl` template is rendered using `envsubst` (available by default on Ubuntu). The deploy script sets environment variables from the manifesto and secrets, then runs `envsubst < config.yml.tpl > config.yml`. It configures pyaleph for testnet use:

- Ethereum chain pointing at the local Anvil instance.
- All other chains disabled.
- Queue topic: `ALEPH-TESTNET`.
- P2P peers: empty (standalone node, no bootstrap needed).
- Storage engine: filesystem.
- IPFS enabled, pointing at the local Kubo container.

## Integration Test Suite

### Runner

pytest, with minimal Python dependencies (`pyyaml`, `pytest`). Tests call the Aleph Rust CLI as a subprocess.

### Configuration

Tests are configured via environment variables:

```
ALEPH_TESTNET_CCN_URL=https://testnet-ccn.example.com
ALEPH_TESTNET_PRIVATE_KEY=0x...
```

`ALEPH_TESTNET_PRIVATE_KEY` is stored as a GitHub Actions secret for the nightly workflow. It is a dedicated test key — no on-chain funding or token balance is needed for phase 1 since posts, files, and aggregates can be created without funds. The same key is reused across runs (stable test identity).

### Fixtures (`conftest.py`)

- **`aleph_cli`** — wrapper function that invokes the CLI binary with `--ccn-url` and `--private-key` pre-configured. Parses JSON output when `--json` is passed (supported by file and message subcommands). For subcommands without `--json`, parses stdout as text. Asserts on exit code.
- **`ccn_ready`** — session-scoped autouse fixture that polls `GET /api/v0/version` on the CCN before any test runs. Retries every 5s for up to 120s. Fails the entire session if unreachable.
- **`tmp_file`** — generates a temporary file with random content for upload tests.

### MVP Test Scenarios (Phase 1)

| Test | What it validates |
|---|---|
| `test_file_upload_and_download` | Upload a file, download by hash, compare bytes |
| `test_file_upload_with_ref` | Upload with `--ref`, download by ref |
| `test_post_create_and_list` | Create a post, list by channel, verify it appears |
| `test_post_amend` | Create a post, amend it, verify updated content |
| `test_aggregate_create_and_read` | Create aggregate with key, read back, verify content |
| `test_message_list` | List recent messages, verify structure |
| `test_message_forget` | Create a post, forget it, verify it disappears |

Each test is independent (no ordering). Tests use unique channels/keys to avoid collisions. Read-after-write operations include a configurable poll/retry to account for CCN processing time.

### Test Execution

- **Nightly workflow** (`nightly.yml`): Runs on `ubuntu-latest`. Downloads the CLI binary from the URL in the manifesto to `./bin/aleph`, `chmod +x`, adds to `$PATH`. Runs `pytest --junitxml=results.xml`, uploads results as GitHub Actions artifacts.
- **Local**: Developers can run `pytest` locally against any CCN by setting the environment variables and ensuring the `aleph` binary is on `$PATH`.

## DigitalOcean Ephemeral Testing (Phase 3)

For PR validation with CRNs:

- Use `doctl` to provision 3-4 droplets (1 main + 2-3 CRNs).
- Droplet size: `s-4vcpu-8gb-amd` for CRNs (Firecracker/QEMU needs 4vCPU + 8GB).
- Region: `ams3` (consistent with aleph-vm's existing setup).
- Deploy the full stack, run tests, tear down.
- Secrets: `DIGITALOCEAN_ACCESS_TOKEN`, `DIGITALOCEAN_SSH_PRIVATE_KEY`.

This is not built in phase 1.

## Dependencies

### Phase 1 Python dependencies (`pyproject.toml`)

- `pytest` — test runner
- `pyyaml` — manifesto parsing

No other dependencies. The CLI is a standalone binary.

### External

- Docker + Docker Compose v2 (plugin, i.e. `docker compose` not `docker-compose`) on the testnet server
- `envsubst` on the testnet server (provided by `gettext-base`, installed by default on Ubuntu)
- `doctl` in GitHub Actions (phase 3 only)
- GitHub Actions runner: `ubuntu-latest` for both workflows
